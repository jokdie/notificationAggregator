# Задача: Notification Aggregator
___
## Контекст

В компании существует несколько независимых сервисов, которые умеют отправлять уведомления пользователям:
- Email Service
- SMS Service
- Push Service

Каждый из них предоставляет собственное HTTP API (после реазизации перевести на gRPC).

Необходимо написать сервис-агрегатор, который принимает один запрос от клиента и самостоятельно рассылает уведомление во все необходимые каналы одновременно.

Бизнес хочет получать итоговую информацию о том, какие каналы успешно приняли уведомление, а какие нет.

При этом внешние сервисы работают нестабильно:
- иногда отвечают медленно; 
- иногда возвращают ошибки; 
- иногда вообще не отвечают.

Сервис должен корректно работать под высокой конкуренцией.
___
## Требования
### Основная функциональность

Реализовать HTTP API, принимающее запрос на отправку уведомления.

После получения запроса сервис должен начать отправку уведомления во все указанные каналы одновременно.

Каждый канал независим.

Ошибка одного канала не должна автоматически означать ошибку остальных.

Результат работы должен содержать статус каждого канала отдельно.

### Внешние сервисы

В рамках задачи необходимо реализовать внутри того же приложения три отдельных HTTP endpoint'а, имитирующих внешние сервисы:
- Email Provider
- SMS Provider
- Push Provider

Каждый сервис должен:

- принимать HTTP запрос;
- поддерживать искусственную задержку;
- уметь возвращать успешный ответ;
- уметь возвращать ошибку.

Конкретный механизм выбора успешного/ошибочного ответа остаётся на твоё усмотрение, но поведение должно быть детерминированным и управляемым.

### Ограничение конкурентности

Во всей системе одновременно разрешено выполнять не более 5 исходящих запросов ко внешним сервисам.

Если лимит достигнут:

- новые запросы должны ждать освобождения места;
- ожидание должно учитывать отмену через Context.

### Timeout

Для каждого обращения к внешнему сервису действует индивидуальный timeout:
**2 секунды.**

Если внешний сервис не успел ответить: канал считается неуспешным.

Остальные каналы продолжают работу.

### Context

Если клиент закрыл соединение:
- все ещё не завершившиеся обращения к внешним сервисам должны быть отменены;
- сервис не должен продолжать выполнять бесполезную работу.

### In-memory cache

Необходимо реализовать кэш статусов отправки.

Если полностью идентичный запрос приходит повторно в течение 30 секунд, повторную рассылку выполнять нельзя.

Необходимо вернуть ранее полученный результат.

TTL — 30 секунд.

После истечения TTL запись считается отсутствующей.

### Background Cleanup

Просроченные записи кэша должны удаляться автоматически в фоне.

Фоновая задача должна корректно завершаться при остановке приложения.

### Graceful Shutdown

По сигналу завершения приложения необходимо:
- прекратить принимать новые HTTP-запросы;
- дождаться завершения уже выполняющихся запросов;
- корректно остановить фоновые задачи;
- не допустить утечек goroutine.
___
## API
### POST /notifications
**Request**
```json
{
  "user_id": 15,
  "message": "Your order is ready",
  "channels": [
    "email",
    "sms",
    "push"
  ]
}
```
**Response 200**
```json
{
  "results": [
    {
      "channel": "email",
      "status": "success"
    },
    {
      "channel": "sms",
      "status": "failed"
    },
    {
      "channel": "push",
      "status": "timeout"
    }
  ]
}
```
### Возможные ошибки
400 - Некорректный JSON.

400 - Неизвестный канал.

400 - Пустой список каналов.

400 - Пустое сообщение.

405 - Неверный HTTP метод.

### Provider API
**POST /provider/email**

**POST /provider/sms**

**POST /provider/push**

Формат входного запроса и ответа можно определить самостоятельно, если он достаточен для решения задачи.
___
## Ограничения
Обязательно соблюдать следующие требования:
- использовать только net/http;
- не использовать сторонние HTTP-роутеры;
- HTTP handler не должен выполнять длительные блокирующие операции сверх необходимых для обработки текущего запроса;
- максимум 5 одновременно выполняющихся исходящих HTTP-запросов ко всем provider'ам суммарно;
- учитывать отмену через context.Context;
- не допускать race condition;
- не допускать deadlock;
- не допускать утечек goroutine;
- все общие структуры данных должны быть потокобезопасны;
- нельзя использовать глобальные изменяемые переменные для хранения бизнес-состояния (за исключением действительно неизменяемых констант);
- приложение должно корректно завершаться по graceful shutdown.
___
## На что обратить внимание
Без подсказок по реализации — только перечень аспектов, которые часто упускают:
- корректная обработка отмены Context;
- работа с таймаутами HTTP-клиента и отдельных запросов;
- потокобезопасность кэша;
- дедупликация полностью идентичных запросов;
- корректность TTL;
- корректное завершение фоновой очистки;
- закрытие ресурсов после HTTP-запросов;
- отсутствие гонок при высокой конкуренции;
- обработка частичных ошибок;
- корректные HTTP-статусы;
- отсутствие утечек goroutine;
- корректное поведение при медленных provider'ах;
- соблюдение общего лимита одновременно выполняемых исходящих запросов;
- корректная работа при 500–1000 RPS (с точки зрения потокобезопасности и ограничения конкурентности, без необходимости проводить нагрузочное тестирование).
___
## Изменение требований (не выполнять сразу)

Открой этот раздел только после завершения основной реализации.
```Через месяц один из провайдеров начинает отвечать за 4–5 секунд и периодически возвращает HTTP 429 (Too Many Requests). Бизнес требует снизить количество неудачных отправок, но среднее время ответа API увеличивать нельзя. Необходимо доработать сервис под новые условия, сохранив существующий контракт API.```
